Core Innovation

Adaptive Lock Elision

Hardware-inspired software technique operating at key-range granularity — eliminating lock overhead when contention is absent, engaging full 2PL safety when it rises.

How It Works

The engine continuously measures a contention ratio per key range using exponential moving averages over CAS failure rates. Under low contention, transactions execute on a CAS-based optimistic path — no lock acquisition, no waiting, zero overhead.

When the ratio crosses a configurable threshold, the Adaptive Lock Manager auto-switches to ReentrantLock 2-Phase Locking for that key range. High-contention ranges get serialization safety; low-contention ranges stay fast.

Inspired by Intel HLE/RTM hardware lock elision, but implemented entirely in software using VarHandle CAS operations and Java 21 Virtual Threads — no hardware transactional memory required.

Technical Deep Dive

Component Internals

Each subsystem engineered for correctness first, then performance — with concurrent data structures at every layer.

01
MVCC with Version Chains
AtomicReference linked list · Epoch-based GC

Each key maintains a version chain — an immutable linked list of versioned values managed via AtomicReference for lock-free prepend. Readers snapshot the chain head at transaction start; concurrent writes append new versions without blocking readers.

Garbage collection uses an RCU-inspired epoch mechanism: each transaction registers in the current epoch on entry and deregisters on commit. A background GC thread, running in a Virtual Thread, reclaims version chain nodes only when no active epoch can reference them — safe, minimal overhead.

// Lock-free version chain prepend private void appendVersion(K key, V value, long txnId) { var newNode = new VersionNode<>(value, txnId); var chain = versionChains.computeIfAbsent(key, k -> new AtomicReference<>()); VersionNode<V> prev; do { prev = chain.get(); newNode.next = prev; } while (!chain.compareAndSet(prev, newNode)); // spin on contention }

02
SSI — Serializable Snapshot Isolation
Dirty read · Write skew · Phantom detection

NexusDB implements Serializable Snapshot Isolation by tracking rw-antidependency edges between concurrent transactions. A transaction T1 has an rw-antidependency on T2 if T1 reads a version that T2 subsequently overwrites.

The SSI Validator detects dangerous structures — cycles of rw-antidependencies of the form T1 →rw T2 →rw T3 →... →rw T1. When such a cycle is detected, the youngest transaction in the cycle is aborted to break it. This handles write skew and phantoms without the performance cost of predicate locks.

// rw-antidependency cycle detection (simplified) boolean formsDangerousStructure(Transaction txn) { var rwIn = antidepIn.getOrDefault(txn, emptySet()); var rwOut = antidepOut.getOrDefault(txn, emptySet()); // Dangerous iff txn has both an incoming and outgoing rw edge return !rwIn.isEmpty() && !rwOut.isEmpty(); }

03
Write-Ahead Log — ARIES Protocol
Group commit · Virtual Threads · Crash recovery

The WAL implementation follows the ARIES (Algorithm for Recovery and Isolation Exploiting Semantics) protocol, maintaining LSN-ordered redo and undo log records for every committed transaction. On crash recovery, the engine performs Analysis → Redo → Undo passes.

Group commit batches multiple concurrent transactions into a single fsync, dramatically reducing I/O overhead. Java 21 Virtual Threads coordinate the batching window — each committing transaction parks on a virtual thread until the batch is flushed, achieving throughput-latency balance without thread pool starvation.

04
Concurrent B-Tree
Hand-over-hand crabbing · Lock-free readers

The storage layer uses a hand-over-hand latch coupling (crab-locking) strategy on the B-Tree. Writers acquire a write latch on a child before releasing the parent — safe against structural modifications like splits and merges. If a node is guaranteed safe (not splitting), the parent latch is released early for maximum concurrency.

Readers use StampedLock optimistic reads: attempt a read without locking, validate the stamp on exit, and retry with a pessimistic read lock only on stamp conflict. Under typical workloads, readers never block writers.

05
Adaptive Lock Manager
Contention monitoring · CAS ↔ 2PL auto-switching

The Adaptive Lock Manager maintains a per-key-range contention ratio using an exponential moving average (EMA) of CAS operation failure rates. A CAS failure is a direct signal of contention — another thread modified the value concurrently.

When the EMA exceeds a configurable threshold (default: 0.3 failures/attempt), the key range is promoted to 2PL mode. Conversely, when contention drops below a hysteresis band, it demotes back to CAS. The switch is atomic and transparent to callers. Timeout-based deadlock detection prevents indefinite blocking in 2PL mode.

// EMA-based contention tracking per key range void recordCasResult(KeyRange range, boolean success) { double sample = success ? 0.0 : 1.0; contentionEma.compute(range, (k, prev) -> prev == null ? sample : EMA_ALPHA * sample + (1 - EMA_ALPHA) * prev ); if (contentionEma.get(range) > PROMOTE_THRESHOLD) promiteTo2PL(range); else if (contentionEma.get(range) < DEMOTE_THRESHOLD) demoteToCas(range); }

Real-World Applications

When to Use NexusDB

01

Embedded Transactional Store

When your application needs an embedded database with full ACID transactions — like SQLite but with concurrent access and MVCC. Think: desktop apps, CLI tools, or microservices that need local persistent state without deploying a separate database server.

02

Hot-Key Workloads

Systems where some keys receive 100x more writes than others — e-commerce inventory counts, rate limiters, leaderboards. NexusDB's adaptive lock elision automatically switches to pessimistic locking only on the hot keys while keeping cold keys lock-free.

03

Financial Transaction Processing

When you need SSI (Serializable Snapshot Isolation) to prevent write skew and phantom reads — banking ledgers, double-entry accounting, trading systems. NexusDB detects anomalies at every isolation level and aborts dangerous transactions automatically.

04

Crash-Safe State Machines

Any system that must survive process crashes without data loss — workflow engines, job schedulers, IoT gateways. The WAL with ARIES redo/undo guarantees committed transactions survive crashes, and recovery is automatic on restart.

05

Read-Heavy Analytics with Concurrent Writes

MVCC lets readers never block writers and vice versa. Ideal for dashboards querying live data, real-time reporting on operational databases, or search indexes that update while serving queries — each reader sees a consistent snapshot without locking.

06

Learning & Reference Implementation

A production-quality reference for database internals — version chains, epoch GC, B-Tree crabbing, WAL recovery, SSI validation. Useful for engineers studying concurrency control, building custom storage engines, or preparing for systems design interviews.
